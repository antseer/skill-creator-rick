# V2 Writing Standard — 基于发布稿 / 模板页的产品化改造写法

> 适用场景：用户提供 `*-v2.html`、发布稿生成页、前端工程师基于模板生成的页面，要求“功能不变，但内部逻辑和视觉效果符合整体产品规范”。

## 1. V2 写法的定义

在 Skill Creator Rick 中，**V2 写法**指一种“产品化改造需求写法”：

- 输入通常是一个已经能跑的 HTML / demo / 发布稿页面。
- 页面可能由前端工程师基于用户发布稿、旧版页面或模板生成。
- 目标不是重新发明功能，而是把它改造成 Antseer 产品规范下的正式页面。

一句话：**V2 写法先锁定已有功能不变，再要求内部逻辑、数据边界、视觉系统和验收方式进入产品规范。**

## 2. 什么时候必须用 V2 写法

出现以下任一情况，应使用 V2 写法：

1. 文件名或需求中出现 `v2.html` / `V2`。
2. 页面已经有可运行版本，但来自旧版改造、模板生成或发布稿二次生成。
3. 用户强调“不要改变功能，只改内部逻辑 / 数据源 / 视觉规范”。
4. 用户担心前端擅自改设计、擅自改数据、引入 mock 或破坏原交互。
5. 目标是把一次性 demo 升级成可验收、可复用、可继续迭代的 Skill 页面。

## 3. 普通写法 vs V2 写法

| 维度 | 普通需求写法 | V2 写法 |
|---|---|---|
| 起点 | 从新功能或 bug 点出发 | 从已有可运行页面出发 |
| 核心问题 | 要新增 / 修复什么 | 哪些功能不能变，哪些实现必须产品化 |
| 对前端的约束 | 说明页面效果 | 明确禁止擅自改功能、数据口径和已确认样式 |
| 数据要求 | 描述需要哪些数据 | 明确真实数据、mock 禁令、缺数据空态和字段合约 |
| 视觉要求 | “优化 UI / 更美观” | 对齐 Antseer Design System，不保留模板感 |
| 验收方式 | 看页面是否能用 | 功能等价 + 数据真实 + 视觉规范 + E2E 可复现 |

## 4. V2 文档必须包含的 10 个部分

### 4.1 来源说明

说明当前页面来自哪里：发布稿、旧版页面、模板生成、前端二次加工等。

### 4.2 改造目标

说明这次不是重做策略，而是产品化改造：功能不变、逻辑分层、视觉规范、数据真实。

### 4.3 功能不变清单

列出所有不可破坏的行为，尤其是用户已经验收过或反复强调过的交互。

### 4.4 允许改造范围

明确可以改：DOM、JS 组织、组件拆分、图表实现、表格边距、空态、tooltip、视觉 token。

明确不可以改：指标定义、数据来源、回测口径、样本规则、免责声明、用户确认过的布局意图。

### 4.5 数据合约

列出页面必须读取的真实字段、文件、MCP/API 来源和前端责任。

硬约束：
- 不允许 mock / random / demo 进入用户主路径。
- 不允许为了视觉连续性补 0、补历史、补事件。
- 缺数据必须显示原因和来源，不得沉默失败。
- 官网正式交付形态是“HTML 模板作为 JSON 响应字段返回”，所以正式模板的渲染数据必须内联在 `#antseer-data` 中；不要把用户主路径绑定到同级 `../data/*.js` 或本地静态文件。
- 内联数据必须是真实 MCP/API/数据库结果的固化快照，并在文档和脚本中说明生成时间、来源和复现方式；内联不等于允许 fixture。

### 4.6 内部逻辑规范

要求分层：

```text
raw data → data adapter → domain calculator → view model → renderer → interaction state
```

renderer 只渲染 view model，不直接请求数据、不生成业务口径、不写 hardcoded 修补。

### 4.7 视觉规范

引用 Antseer Design System：暖黑背景、卡片、边框、字体、正负色、金融图表、两栏 input-only/output-only。

### 4.8 交互与图表规范

写清主图、附图、marker、拖动缩放、多选、事件列表、tooltip 和空态。

### 4.9 可复用组件沉淀

如果页面包含 K 线、附图、marker、事件 rail、数据 inspector，应定义组件输入合约，方便沉淀进 Skill factory / SQL factory。

### 4.10 验收门禁

必须能被同事复现：
- no mock / no random。
- 数据字段存在性检查。
- 核心交互 E2E。
- 视觉规范 review。
- 样本量、时间范围、数据来源、免责声明可见。

## 5. 推荐模板

```markdown
---
status: active
updated: YYYY-MM-DD
description: {Skill Name} {Version} V2-style 产品化改造需求
skills: skill-creator-rick
phase: {Version}
---

# {Skill Name} · {Version} V2-style Requirement

## 1. 来源说明
## 2. 改造目标
## 3. 功能不变清单
## 4. 允许改造范围
## 5. 数据合约
## 6. 内部逻辑规范
## 7. 视觉规范
## 8. 交互与图表规范
## 9. 可复用组件沉淀
## 10. 验收门禁
```

## 6. 合格标准

一份合格的 V2 写法文档，必须让前端工程师清楚：

1. 当前页面哪些功能不能动。
2. 哪些模板痕迹和内部逻辑债必须清掉。
3. 数据从哪里来，缺了怎么办。
4. 视觉应该对齐哪套产品规范。
5. 如何证明自己没有引入 mock、破坏交互或篡改数据口径。
